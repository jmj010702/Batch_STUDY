# 레디스 분산락 VS Quartz 스케줄러 

--- 

둘을 비교하기에 앞서 먼저 둘은 목적 자체가 다르다 뭐가 다른가 

                        redis 분산락                              Quartz Schedular 
                        
본질         |        동시성 제어 도구           |           작업 예약/실행 엔진 
핵심 질문     |        지금 이 작업을 누가 실행해도 되는가  |   언제, 어떤 작업을 어떻게 실행해야 하나
상태 저장     |      Lock 점유 여부만           |           Job,Trigger 실행 이력 전체 (스케줄러의 전반적인 상태)
실패 복구     |        없음(TTL 만료후 해제)      |          misfire 정책으로 재실행 가능 
모니터링       |     직접 구현 필요               |           내장 JobListener, TriggerListener

---

redis 분산락만 사용했을 때의 한계
1. 스케줄링 기능이 전혀 없다  때문에 @Scheduled(cron) 으로 무조건 조합을 해야함 -> @Scheduled는 각 인스턴스마다 독립적으로 실행되기
 때문에 락이 없으면 3개 인스턴스(서버)가 동시에 실행이 됨. 락이 있어도 언제 실행될지는 하드 코딩됨 (동적으로 바꿀 수 있는 방법은 복잡하지만 있음)

2. 서버가 끊길 시 끊긴 진행 상황에 대해서 재 실행 여부가 없다
- 1. 만일 A서버가 락 획득 { redis에 TTL은 10초 }
  2. A서버 다운
  3. 나머지 다른 서버들은 이미 락을 보고 죽은 작업들이기 때문에 실행이 안됨
  4. A서버가 하던 나머지 일들은 없는 아무 일도 일어나지 않음

 - quartz는 misfire 정책에 따라 재실행 여부를 설정할 수 있음

3. 실패한 작업의 재실행(MisFire) 처리 불가
  - 1. 시작하기 전에 서버가 죽든 실행하다가 서버가 죽어도 해당 작업에 대해 재실행 가능성이 없음. 만일 처리를 해주려면 코드로 다른 처리를 해줘야 할 것
   
  - 위에 정책과 같음 

4. 실행 이력 모니터링이 불가능함
   - 1. redis 분산락 만으로는 알 수 있는 사실이 현재 락이 걸려있나 없나 뿐 만일 실행 이력을 알고 싶다면 일일히 로그를 찍어줘여 함

   - db에 진행 사항을 기록하기 때문에 모니터링이 수월함
